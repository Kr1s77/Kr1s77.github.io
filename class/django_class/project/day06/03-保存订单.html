    def create(self, validated_data):
        # 1. 获取参数 address 和 pay_method
        address = validated_data['address']
        pay_method = validated_data['pay_method']

        # 获取当前下单用户
        user = self.context['request'].user

        # 2. 组织参数:
        # 订单id:
        # 2.1 格式:  '年月日时分秒' + 用户id
        # datetime.now().strftime('%Y%m%d%H%M%S') ==> '20180904120410'
        # 拼接上10位的字符串(不够10位的补0)
        order_id = datetime.now().strftime('%Y%m%d%H%M%S') + ('%010d' % user.id)

        # 订单状态: 
        if pay_method == 1:
            # 货到付款: 
            status = 2  # 待发货:
        else:
            # 在线支付: 
            status = 1  # 待支付

        # 4.订单商品总数量和实付款
        total_count = 0
        total_amount = Decimal(0)
        # 运费: 
        freight = Decimal(10)

        #  3. 向订单基本信息表添加一条记录:
        order = OrderInfo.objects.create(
            order_id = order_id,
            user = user,
            address = address,
            total_count = total_count,
            total_amount = total_amount,
            freight = freight,
            pay_method = pay_method,
            status = status
        )

        # 订单中包含几个商品, 就需要向订单商品表中添加几条记录
        # 获取redis链接: 
        redis_conn = get_redis_connection('cart')

        # 从redis中获取用户所要购买的商品的sku_id 
        sku_ids = redis_conn.smembers('cart_selected_%s' % user.id)

        #{
        #    b'<sku_id>':b'<count>',
        #    ...
        #}
        cart_dict = redis_conn.hgetall('cart_%s' % user.id)

        for sku_id in sku_ids:
            # 获取用户所要购买的该商品的数量 count
            count = cart_dict[sku_id]
            count = int(count)

            # 根据 sku_id 获取商品信息:
            sku = SKU.objects.get(id=sku_id)

            # 减少商品库存, 增加销量
            sku.stock -= count
            sku.sales += count 
            sku.save()

            # 向订单商品表中添加一条记录
            OrderGoods.objects.create(
                order = order,
                sku=sku,
                count = count,
                price = sku.price
            )

            # 累加计算订单中商品总数量和总金额:
            total_count += count
            total_amount += sku.price * count

        # 实付款: 
        total_amount += freight
        # 更新订单记录中商品的总数量和实付款
        order.total_count = total_count
        order.total_amount = total_amount
        order.save()


        # 清除购物车中对应的记录: 
        redis_conn.hdel('cart_%s' % user.id, *sku_ids)
        redis_conn.srem('cart_selected_%s' % user.id, *sku_ids)

        # 返回订单对象:
        return order

我们来看: 事务

数据库事务

在保存订单数据中，涉及到多张表（OrderInfo、OrderGoods、SKU）的数据修改，对这些数据的修改应该是一个整体事务

即, 要么一起成功，要么一起失败

mysql 事务定义:

一组 mysql 语句, 放在同一事务中, 要么都成功, 要么都失败.

mysql 事务基本操作:

  • begin 或 start transaction:

    手动开启一个 mysql 事务.

  • commit :

    事务提交, 让事务中 sql 语句的执行结果永久有效.

  • rollback:

    事务回滚, 撤销事务中 sql 语句的执行结果.

在 mysql 事务中, 可以设置事务保存点, 在进行事务语句回滚时可以只回滚到某个保存点.

  • savepoint 保存点名称:

    设置 mysql 事务保存点

  • rollback to 保存点名称:

    回滚事务语句到保存点的位置, 保存点之后 sql 语句的执行结果会被回滚撤销 .

Django 中对于数据库的事务使用:

默认每执行一句数据库操作，便会自动提交

我们需要在保存订单中自己控制数据库事务的执行流程。

在 Django 中:

可以通过 django.db.transaction 模块提供的 atomic 来定义一个事务， atomic 提供两种用法

  • 装饰器用法
from django.db import transaction

@transaction.atomic
def viewfunc(request):
    # 这些代码会在一个事务中执行
    .......
  • with 语句用法
from django.db import transaction

def viewfunc(request):
    # 这部分代码不在事务中，会被Django自动提交

    # 增加事务: 
    with transaction.atomic():
        # 在with语句库中代码, 凡是涉及到数据库操作的代码, 
        # 都会放到同一个事务中, 如果这里的语句最终没有错, 则会被自动提交.
        # 如果最终出错了, 则会自动回滚.


        # 创建保存点
        save_id = transaction.savepoint()  



        # 回滚到保存点
        transaction.savepoint_rollback(save_id)

在 Django 中，还提供了保存点的支持，可以在事务中创建保存点来记录数据的特定状态，数据库出现错误时，可以恢复到数据保存点的状态

并发处理

在多个用户同时发起对同一个商品的下单请求时

先查询商品库存，再修改商品库存，会出现资源竞争问题，导致库存的最终结果出现异常

解决办法：

  • 悲观锁

    当查询某条记录时，即让数据库为该记录加锁，锁住记录后别人无法操作，使用类似如下语法

    # 在查询语句中增加了 for update
# 语句意义: 在获取数据的时候, 对数据加锁: 
select stock from tb_sku where id=1 for update;
    django中悲观锁的使用:
    # 由原来的这个语句: 
sku = SKU.objects.get(id=sku_id)

# 改为现在的: 
# 即: 获取数据的时候, 增加了悲观锁:
sku = SKU.objects.select_for_update().get(id=sku_id)

    悲观锁类似于我们在多线程资源竞争时添加的互斥锁，容易出现死锁现象，采用不多

  • 乐观锁

    乐观锁并不是真实存在的锁，而是在更新的时候判断此时的库存是否是之前查询出的库存，

    如果相同，表示没人修改，可以更新库存

    否则表示别人抢过资源，不再执行库存更新

    类似如下操作:
    # 相比原来增加了 stock = 7 的判断
# 即, 判断当前的库存数量和查询时候的库存数量是否相等
update tb_sku set stock=2 where id=1 and stock=7;
    django中乐观锁的使用:
    # 由原来的这个语句: 
sku = SKU.objects.get(id=sku_id)

# 改为现在的: 
# 即: 获取数据的时候, 增加了乐观锁:
# 注意: update() 这个函数返回的是一个 num 类型的值, 代表更新的行数.
# 更新一条记录, num值为1, 如果一条记录都没有更新, 则为0
num = SKU.objects.filter(id=sku_id, stock=7).update(stock=2)

# 补充: 
如果更新失败, 则代表库存发生了改变, 不代表库存不足.
我们会重新进行尝试, 最多试3次

  • 任务队列

    将下单的逻辑放到任务队列中（如celery），将并行转为串行，所有人排队下单。比如开启只有一个进程的Celery，一个订单一个订单的处理。

修改 MySQL 的事务隔离级别

事务隔离级别指的是在处理同一个数据的多个事务中，一个事务修改数据后，其他事务何时能看到修改后的结果。

MySQL 数据库事务隔离级别主要有四种：

  • Serializable

    串行化级别

    • 一个事务一个事务的执行

  • Repeatable read

    可重复读级别

    • 无论其他事务是否修改并提交了数据，在当前事务中看到的数据值始终不变, 读取的仍是同样的数据
    • mysql 使用的是该模式, 其他数据库不是.

  • Read committed

    读取已提交的数据级别

    • 其他事务提交了对数据的修改后，本事务就能读取到修改后的数据值
    • 除了 mysql 以外, 其他数据库默认使用的都是该模式.

  • Read uncommitted

    读取未提交的数据

    • 其他事务只要修改了数据，即使未提交，本事务也能看到修改后的数据值

MySQL 数据库默认使用可重复读（ Repeatable read），而使用乐观锁的时候，如果一个事务修改了库存并提交了事务，那其他的事务应该可以读取到修改后的数据值，所以不能使用可重复读的隔离级别，应该修改为读取已提交Read committed。

Mac下事务隔离级别的查看与修改:

第一步: 进入 mysql
# 首先, 需要进入 mysql:
mysql -uroot -p123456
第二步: 查看 mysql 中事务级别的名称
# 敲如下命令能够查看 mysql 中事务配置的各个名称: 
show variables like '%isolation%';

结果:

第三步: 查看 mysql 中事务级别的名称
# global.transaction_isolation:  是全局的事务隔离级别，
# transaction_isolation:    是当前会话的事务隔离级别
select @@global.transaction_isolation, @@transaction_isolation;

结果:

第四步: 将其修改为read committed

以前为 repeatable read

# 修改全局事务隔离级别  
set global transaction isolation level read committed;

# 修改当前会话的事务隔离级别
set session transaction isolation level read committed;

结果:

select @@global.transaction_isolation, @@transaction_isolation;
